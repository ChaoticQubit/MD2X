# md2x site — evaluation harness

Evals for the **`md2x site`** AI generator, organized into three tiers. The
north star (Thariq): `md2x site` must *transform* a Markdown document into a
polished, living website — render each piece of information in the shape it
actually has — not merely re-render the prose. These evals measure how well it
does that.

## The three tiers

| Tier | What it checks | Cost | Where it lives | Runs in CI? |
|------|----------------|------|----------------|-------------|
| **Tier-1 — deterministic** | Structural invariants of the render layer: blocks → DOM mapping, escaping/sanitization, self-contained output, no network refs. Pure, fast, no model. | Free | `tests/evals/` | **Yes** |
| **Tier-2 — quality (LLM-as-judge)** | Whether a generated page is a genuine living site, scored against the Thariq rubric by an LLM judge. | **Tokens** | **here** (`evals/`) | **No — opt-in** |
| **Tier-3 — performance** | Runtime + peak memory of the deterministic build+render path (the floor every run pays). | Free (no model) | **here** (`evals/`) | **No — opt-in** |

Tier-1 is the deterministic assertion suite: it is fast and free, so it runs in
CI on every change. Tier-2 and Tier-3 are the contents of this directory; they
are **opt-in** because Tier-2 spends tokens / needs a real model, and we keep
both out of CI so the pipeline stays fast and free.

## Running them (opt-in)

Both scripts are inert unless `MD2X_RUN_EVALS` is set — importing them, or
running them without the flag, does nothing expensive and prints a hint:

```bash
# Tier-2: quality / LLM-as-judge (spends tokens — needs a real model + API key)
MD2X_RUN_EVALS=1 python evals/run_quality_eval.py

# Tier-3: performance (no model required, still gated to stay out of CI)
MD2X_RUN_EVALS=1 python evals/run_performance_eval.py
```

> **Token-cost warning.** `run_quality_eval.py` generates a page for each
> reference document **and** invokes an LLM judge to score it. That is multiple
> model calls per run and **costs real tokens / money**. It is opt-in by design
> and must never run in CI or on import.

The quality eval exits non-zero (FAIL) if the overall average judge score falls
below the threshold (`7.0` / 10), so it can gate a manual quality check; the
performance eval prints runtime/memory stats.

## Provider-agnostic

Nothing here hardcodes a provider or SDK. The judge model is built through the
project factory `md2x.site.models.build_model(ai_cfg, role)`, driven by an
`ai_cfg` dict (e.g. `{"model": "anthropic:claude-sonnet-4-6", "retries": 2}`).
Point it at any provider — a native `provider:model_id` string or an
OpenAI-compatible / local endpoint — via your `md2x.yaml` (the `ai:` block) or
environment, exactly like the real `md2x site` command. The reference default is
just a default.

## Files

- `thariq_rubric.py` — `RUBRIC` + `DIMENSIONS`: the five scored dimensions
  (shape, density/scannability, faithfulness, interactivity, on-brand &
  self-contained) and what 0 vs 10 looks like for each. The single source of
  truth both scripts import. Pure data — no agno, free to import.
- `run_quality_eval.py` — Tier-2. For each reference doc: build a `Doc` from the
  raw Markdown (no pandoc), generate the page through the real md2x agents
  (blocks/full), then score it with an agno `AccuracyEval` whose judge reasons
  through `RUBRIC`. Prints a per-doc + overall table and PASS/FAIL.
- `run_performance_eval.py` — Tier-3. Wraps the deterministic
  `build_page_doc` + `render_blocks` path (no model) in an agno `PerformanceEval`
  and reports runtime + peak memory.
- `reference/` — four realistic, self-contained source documents that exercise
  different archetypes and are the inputs the evals generate from:
  - `report.md` — a weekly status report with KPIs, highlights, risks, decisions.
  - `explainer.md` — how the edge cache works: request path, config variants, FAQ.
  - `deck.md` — a short talk outline, one idea per section.
  - `editor.md` — a launch-readiness triage board that wants a custom editor.

## How Tier-2 works (briefly)

agno's `AccuracyEval.run()` produces the answer it judges by calling
`agent.run(...)`. The page has already been generated by the real md2x pipeline,
so the eval wraps that finished HTML in a tiny fixed-response agent that just
returns it; the judge model then scores **the true md2x output** against the
rubric. All agno / model imports live inside functions — these scripts import
cleanly even without the `[ai]` extra installed.
